Why this isn't just another RAG demo

A "simple RAG" pipeline is a single straight line: embed the question, grab the top-k chunks, stuff them in a prompt, generate, done. It has no opinion about whether retrieval was even needed, whether the chunks it grabbed were actually relevant, or whether the answer it wrote is actually backed by them. This graph adds four checkpoints a simple pipeline skips entirely:

Failure mode in simple RAGHow this graph catches it
Retrieves docs even for questions that don't need themdecide_retrieval routes general-knowledge questions straight to a direct answer — zero wasted retrieval calls
Blindly trusts whatever the vector search returns, even off-topic chunksis_relevant filters retrieved chunks before they ever reach the prompt
Hallucinates details not actually present in the retrieved contextis_sup checks the draft against the context and loops into revise_answer until it's grounded
Returns a technically-grounded but unhelpful non-answeris_use checks the answer actually addresses the question, and loops into rewrite_question to retry retrieval with better keywords
Fabricates a confident-sounding answer when nothing relevant existsno_answer_found returns an honest "not in these documents" instead of guessing

Simple RAG vs this Adaptive RAG

A simple RAG pipeline is one straight line with no way to know when it's wrong. This graph checks itself at four points and loops back when something's off.

Simple RAG no self-correction

  • Always retrieves, even for general knowledge
  • Never filters irrelevant chunks
  • Never checks if the answer is grounded
  • Never checks if the answer is actually useful
  • Can't recover — one shot, no loops

This graph self-correcting

  • Skips retrieval for general-knowledge questions
  • Filters out off-topic chunks before generation
  • revise_answer loop fixes unsupported claims
  • rewrite_question loop retries with better keywords
  • Says "I don't know" instead of fabricating an answer
